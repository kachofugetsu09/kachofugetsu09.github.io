# MySQL 核心概念笔记

# SQL 语句的执行逻辑

> 🔑 **核心要点**：SQL语句在MySQL中的执行是一个多阶段协作的过程，涉及连接器、解析器、优化器等多个组件，旨在确保安全、高效、准确地处理数据。

## 是什么

SQL 语句在 MySQL 数据库中的执行流程可以概括为以下核心组件的协同工作：

1.  **连接器 (Connector)**：负责客户端与数据库的**连接建立、身份认证和会话管理**，维护连接的权限及状态。
2.  **查询缓存 (Query Cache)**：**（MySQL 8.0 之前）** 缓存完整的 SQL 语句及其结果集，相同查询可直接返回缓存。
3.  **解析器 (Parser)**：对 SQL 语句进行**词法分析（拆分 Token）**和**语法分析（生成抽象语法树 AST）**，检查语法错误。
4.  **预处理器 (Preprocessor / Semantic Analyzer)**：在语法正确基础上，进行**语义校验**，如检查表/列是否存在及权限，并进行类型推导。
5.  **优化器 (Optimizer)**：SQL 执行的**"大脑"**。根据成本模型、统计信息，分析 AST，**生成并选择最优的执行计划**（如选择索引、连接顺序等）。
6.  **执行器 (Executor)**：SQL 执行的**"执行者"**。根据优化器选定的执行计划，调用底层存储引擎接口执行具体的数据操作（读写、过滤、排序等），并返回结果。

## 为什么

> 💡 **设计理念**：每个阶段都承载着特定的关键职责，共同确保数据库的安全性、稳定性、高效性和可扩展性。

1.  **连接器的重要性**：作为数据库**安全的第一道防线**，通过身份验证和权限管理保障访问安全。同时，它管理连接的会话上下文，为后续事务隔离等提供基础。
2.  **查询缓存的演进**：MySQL 8.0 移除查询缓存，核心原因是其**粗粒度的失效机制**。任何数据修改都会导致相关缓存大规模失效，在高并发和数据频繁更新场景下，**命中率极低**，反而引入额外的**维护开销和锁竞争**，成为性能瓶颈，体现了数据库设计中对**实际性能权衡**的考量。
3.  **解析器的职责**：将人类可读的 SQL **翻译成机器可理解的结构化指令（AST）**，是所有后续处理的基础。其**早期语法检查**能及时发现错误，避免资源浪费。
4.  **预处理器的作用**：弥补纯语法检查的不足，确保 SQL 语句在逻辑上 **"有意义且合法"** ，防止操作不存在的对象或越权访问，是保障数据安全的必要步骤。
5.  **优化器的核心地位**：优化器是数据库性能的 **"决策核心"** 。它将声明式的 SQL 语句转化为高效的操作序列，通过精密的成本评估（基于数据量、索引、扫描方式等），**旨在最小化磁盘 I/O 和 CPU 消耗**。一个高质量的执行计划对查询性能至关重要，是 SQL 调优的关键。
6.  **执行器的实现**：将优化器的"蓝图"转化为实际操作，与底层存储引擎紧密协作。其效率直接受优化器计划质量和存储引擎能力的影响。

# MySQL 的存储引擎

> 🔑 **核心要点**：MySQL 支持多种存储引擎，其中 InnoDB 是最重要的事务型存储引擎，而 MyISAM 则适用于特定的非事务性场景。

## 是什么

MySQL 的存储引擎是数据库管理系统的核心组件，负责数据存储、索引、事务管理、并发控制、崩溃恢复等底层操作。

1.  **InnoDB (默认，事务型)**：
    * **核心特性**：全面支持 **ACID 事务**；采用**行级锁**实现高并发；支持**外键约束**；具备强大的**崩溃恢复能力**（通过 Redo Log 和 Undo Log）。
    * **索引机制**：实现**聚簇索引**，主键索引即数据本身；所有**辅助索引**的叶子节点存储的是主键值，需要**"回表"**才能获取完整数据。
    * **适用场景**：绝大多数需要**高并发、强事务一致性**和数据完整性的 OLTP (在线事务处理) 应用。

2.  **MyISAM (非默认，非事务型)**：
    * **核心特性**：**不支持事务、行级锁和外键**；只支持**表级锁**导致高并发写入差；不支持崩溃恢复。
    * **索引机制**：全部采用**非聚簇索引**，数据文件（.MYD）与索引文件（.MYI）**完全分离**；索引叶子节点直接存储**数据行物理地址**，**无需"回表"**。
    * **额外特性**：支持**全文索引**；维护精确行数计数器，`SELECT COUNT(*)` 极快。
    * **适用场景**：对数据一致性要求不高、以读操作为主且并发写入极少的场景，如数据仓库中的某些静态分析表。

3.  **Memory (内存型)**：
    * **核心特性**：数据完全存储在**内存**中，访问极快；数据**不持久化**（服务器重启丢失数据）；不支持事务、行级锁和外键。
    * **适用场景**：临时表、高速缓存、数据分析过程中的中间结果集等对数据持久性无要求的场景。

## 为什么

> 💡 **InnoDB 成为默认引擎的根本原因**：它在数据一致性、完整性、崩溃恢复和高并发性方面提供了更全面的保障，更符合现代企业级应用的需求。

1.  **事务支持**：InnoDB 全面支持 **ACID 事务**，从根本上保证了数据在复杂操作和系统故障下的**一致性与完整性**，这是大多数关键业务系统不可或缺的。
2.  **并发性能**：通过**行级锁**，InnoDB 能够将锁的粒度降到最低，显著减少锁冲突，从而在多用户**高并发读写**场景下提供出色的吞吐量，远优于 MyISAM 的表级锁。
3.  **崩溃恢复**：InnoDB 具备强大的**崩溃恢复能力**，通过 **Redo Log**（保障持久性，前滚已提交事务）和 **Undo Log**（保障原子性，回滚未完成事务）机制，即使在系统突然崩溃后也能保证数据不丢失和事务的正确性，这是 MyISAM 无法比拟的。

# 数据文件的存储结构与索引

> 🔑 **核心要点**：数据文件的存储结构和索引是密不可分的整体。理解它们的物理组织方式，是掌握MySQL性能优化的基础。

## 数据文件的存储结构

### 是什么

MySQL 数据库在文件系统上的存储结构远不止简单的数据文件，它是一个精心组织的体系，反映了数据库引擎的功能和数据持久化的需求。

db.opt (数据库选项文件)：每个数据库目录下有一个，存储该数据库的默认字符集、排序规则等配置信息。

tablename.frm (表结构定义文件)：每个表都有一个，独立于存储引擎，记录表的列定义、数据类型、索引定义等元数据。这是 SQL CREATE TABLE 语句的产物。

InnoDB 相关文件：

tablename.ibd (表数据文件)：当 innodb_file_per_table=ON (默认设置) 时，每个 InnoDB 表的数据和所有索引（包括聚簇索引和所有辅助索引）都存储在一个独立的 .ibd 文件中。这些文件内部以页 (Page) 为基本单位（通常 16KB）进行管理，B+树的每个节点都对应着一个或多个页。

ib_logfileX (Redo Log 文件)：这些文件记录了所有对数据进行的修改操作（事务日志），用于保证事务的持久性 (Durability) 和崩溃恢复。在系统崩溃后，MySQL 会通过这些日志将未提交的事务回滚，已提交但未写入磁盘的数据重新应用。

ibdataX (共享表空间文件)：在 innodb_file_per_table=OFF 或存储 InnoDB 系统表、Undo Log、双写缓冲区等系统级数据时使用。

MyISAM 相关文件：

tablename.MYD (My Data 文件)：存储 MyISAM 表的实际数据行，数据按插入顺序物理存储，无特定排序。

tablename.MYI (My Index 文件)：存储 MyISAM 表的所有索引信息，是独立于数据文件存在的。

### 为什么

这种文件组织结构是为了实现数据库的核心功能、性能优化和健壮性。

**元数据与数据分离**：`.frm` 文件独立存储表结构，使得数据库可以快速加载表定义，并且存储引擎可以灵活选择。

**页式管理**：数据库系统不直接操作文件中的单个字节，而是以固定大小的"页"为单位进行磁盘 I/O。这与操作系统内存管理中的页概念相呼应，能高效利用磁盘的顺序读写特性和局部性原理，一次 I/O 就能读取整个数据块，减少寻址时间。这是实现 B+ 树高效查找的基础。

**InnoDB 的事务与恢复体系**：`ib_logfileX` (Redo Log) 和 Undo Log (包含在 ibd/ibdata 中) 是 InnoDB 实现 ACID 事务和崩溃恢复的关键。Redo Log 记录了数据修改的物理操作，用于前滚（将已提交但未写入磁盘的数据应用回来）；Undo Log 记录了事务的逆操作，用于回滚未完成或需要撤销的事务。这种设计分离了日志记录与实际数据页的写入，提高了写操作的性能，并保证了在系统故障时数据的完整性。

**MyISAM 的简洁与局限**：数据和索引的物理分离（`.MYD` 和 `.MYI`）使得 MyISAM 的结构相对简单，适合快速的全表扫描和 `COUNT(*)` 操作。但这种简洁性也带来了无法支持事务、表级锁和低崩溃恢复能力的局限性。

**独立表空间** (`.ibd` 文件)：将每个表的数据和索引存储在独立的 `.ibd` 文件中，方便了表级别的备份、恢复、碎片整理和传输，也使得可以更好地进行空间管理和回收。

## 索引

### 是什么

索引是数据库中用于显著加速数据检索的特殊数据结构，它通过预先排序或组织特定列的数据，创建数据记录与存储位置的映射，从而将全表扫描（线性查找）转变为高效的索引查找。索引的分类基于其数据结构、物理存储方式、字段特性和字段数量。

**按数据结构**：

*B+ 树索引*：最常用和高效的索引结构，广泛应用于关系型数据库。其特点是多路平衡树，所有数据（或数据指针）仅存在于叶子节点，叶子节点之间通过双向链表连接，非常适合范围查询。

*哈希索引*：通过哈希函数将键值映射到存储位置。查询速度理论上为 O(1)，但仅适用于等值查询，不支持范围查询和模糊匹配，且存在哈希冲突问题。

*全文索引*：专门用于文本内容的模糊查询，如 `LIKE '%keyword%'`，基于倒排索引实现，能高效进行关键词搜索。

**按物理存储**：

*聚簇索引* (Clustered Index)：数据的物理存储顺序与索引键值顺序一致。在 InnoDB 中，通常是主键索引。它不仅包含键值，叶子节点还直接存储完整的行数据。一个表只能有一个聚簇索引。

*非聚簇索引* (Non-Clustered Index)：索引的物理存储独立于数据存储。索引叶子节点存储的是键值和指向实际数据位置的指针（在 InnoDB 中是主键值，在 MyISAM 中是物理地址偏移量）。一个表可以有多个非聚簇索引（辅助索引）。

**按字段特性**：

*主键索引* (Primary Key Index)：特殊的唯一非空索引，每张表最多一个。

*唯一索引* (Unique Index)：索引列的值必须唯一，但可以包含 NULL。

*普通索引* (Normal/Secondary Index)：最基本的索引，无额外限制。

*前缀索引* (Prefix Index)：对字符串类型列的部分前缀创建索引，节省空间，但可能降低查询精度和优化能力。

**按字段数量**：

*单列索引* (Single-Column Index)：只对表中的一个列创建索引。

*联合索引* (Composite/Compound Index)：对表的多个列组合创建索引，遵循"最左前缀原则"。

### B+树索引的特性与实现

B+ 树是一种高度优化的、自平衡的多路查找树，它是关系型数据库（如 MySQL InnoDB）中最核心和广泛应用的索引数据结构。它从 B 树演变而来，但对其进行了关键改进，以更好地适应磁盘存储的特性和数据库查询的需求。

**主要特性**：

*多路分支*：每个节点可以有多个子节点（通常数百甚至上千个），这使得树的高度极低。

*键值冗余与数据全在叶子节点*：非叶子节点（内部节点）只存储键值（用于导航）和指向子节点的指针，不存储实际的数据记录或数据指针。所有真实的数据记录或指向数据记录的指针都只存储在叶子节点中。

*叶子节点链表*：所有叶子节点构成一个有序的双向链表。这种设计极大地便利了范围查询和全表扫描（虽然一般不用全表扫描索引）。

*节点与磁盘页匹配*：每个 B+ 树节点的大小通常设计为与数据库的磁盘页（Page）大小（如 16KB）保持一致。

### 索引覆盖 (Covering Index) 是如何避免回表的？

**是什么：** 索引覆盖是指在一次查询中，如果**查询所需的所有列都包含在辅助索引的叶子节点中**（即查询的字段全部是索引列或者索引列的子集），那么执行计划就**无需回表**去聚簇索引中获取完整的数据行。

**如何避免回表：**
* 当查询只涉及索引列时，MySQL 只需要遍历辅助索引的 B+树，就能直接从**辅助索引的叶子节点**中获取到所需的所有数据。
* 辅助索引的叶子节点中已经包含了**索引列的值和主键值**。如果查询的 `SELECT` 列表只包含索引列和主键列，那么在找到辅助索引的叶子节点后，就没有必要再通过主键值去聚簇索引中查找完整行数据，从而避免了额外的磁盘 I/O 操作（回表）。

**示例：**
假设表 `users` 有 `id` (主键), `name`, `age`, `city` 列。
我们为 `(name, age)` 创建了一个联合辅助索引 `idx_name_age`。

1.  **需要回表的查询：**
    `SELECT name, age, city FROM users WHERE name = 'Alice';`
    * 这个查询需要 `city` 列，而 `city` 不在 `idx_name_age` 索引中。
    * 所以，通过 `idx_name_age` 找到 `name='Alice'` 的记录后，还需要**回表**到聚簇索引中获取 `city` 列的值。

2.  **索引覆盖的查询（无需回表）：**
    `SELECT name, age FROM users WHERE name = 'Alice';`
    * 这个查询只涉及 `name` 和 `age` 两列，这两列都包含在 `idx_name_age` 索引中。
    * MySQL 可以直接通过遍历 `idx_name_age` 索引的叶子节点，获取到 `name` 和 `age` 的值，**无需回表**。
    * `EXPLAIN` 结果中可能会显示 `Extra` 字段有 `Using index`。

3.  **索引覆盖的查询（包含主键，也无需回表）：**
    `SELECT id, name, age FROM users WHERE name = 'Alice';`
    * 即使查询中包含了主键 `id`，由于辅助索引的叶子节点本身就存储了主键值，所以这个查询也属于索引覆盖，**无需回表**。

**优点：**
* **减少磁盘 I/O：** 避免了回表操作，减少了随机 I/O，这是性能提升的关键。
* **提高查询效率：** 查询只需扫描辅助索引，通常比扫描整个聚簇索引要小得多，效率更高。

**实现索引覆盖的策略：**
* 根据实际查询需求，创建包含所有查询列的**联合索引**。
* `SELECT` 语句只选择需要的列，避免 `SELECT *`。

# 日志系统

> 🔑 **核心要点**：MySQL的日志系统由Redo Log、Undo Log和Binlog三大日志协同工作，共同保证了事务的ACID特性和数据库的可靠性。

### 是什么

RedoLog： 重做日志，是InnoDB特有的日志，他是一种物理日志。
记录了所有对数据页的物理修改比如"在表 X 的数据页 Y 上，偏移量 Z 处的值从 A 变成了 B"。这些日志以顺序追加的方式写入磁盘，形成一个循环写入的日志文件组。
UndoLog： 回滚日志，也是InnoDB特有的日志，他是一种逻辑日志。
他记录了数据在被修改之前旧版本的值，比如说"在表 X 的数据页 Y 上，偏移量 Z 处的值从 A 变成了 B，那么在回滚时就需要将这个值改回 A"。发控制）。
BinLOG： 是MYSQL自己的二进制日志，他是一种逻辑日志。
记录了所有对数据库的逻辑修改操作，比如"在表 X 上执行了 INSERT 操作，插入了值 A"。
### 为什么
RedoLog使用WAL机制让数据库在事务提交时即使数据还没来得及从内存刷盘就发生崩溃也能在重启后将这些已提交的事务重做回来。WAL是一个预写式日志，思路就是对数据库的任何数据页做修改之前必须先将这些修改操作记录到持久化的日志当中，并且这些日志记录要优先于实际数据页被刷新到磁盘当中。在事务提交前，首先写日志，然后持久化，再去做事务提交，这样保证了所有提交的事务一定是被持久化了的。持久化了但是没提交的事务是可以被恢复的。
UndoLog提供了事务回滚能力，无论事务执行到那个阶段都能用UndoLog回滚到上一个状态。
BinLog使得数据可以进行某一个时间点的恢复，出现错误操作可以通过精确到某个时间的数据恢复。同时，BinLog也可以用于主从复制，主库的所有操作都会记录到BinLog中，从库可以通过读取主库的BinLog来进行数据同步。

# 事务

> 🔑 **核心要点**：事务是数据库的基础功能单元，通过ACID特性保证数据的一致性和可靠性。理解事务的特性和实现机制，是深入理解MySQL的关键。

### 是什么

是一个数据库操作的逻辑单元，它由一系列的数据库操作组成，这些操作要么全部成功，要么全部失败。事务的四个特性被称为ACID特性：
原子性 (Atomicity)：事务中的所有操作要么全部执行成功，要么全部撤销。即使在系统崩溃或错误发生时，也能保证事务的完整性。  
一致性 (Consistency)：事务执行前后，数据库的状态必须保持一致。即事务的执行不会破坏数据库的完整性约束。
隔离性 (Isolation)：多个事务并发执行时，每个事务都应该像是
独立执行的，互不干扰。即使多个事务同时操作同一数据，也不会影响各自的结果。
持久性 (Durability)：一旦事务提交，其结果是永久性的，即使系统
崩溃或重启，已提交的事务结果也不会丢失。
### 为什么
可以防止部分更新，有些操作比如说扣款和扣减库存，这两个操作要么都成功，要么都失败，不能出现扣款成功但库存没有减少的情况。

# 并发问题与事务隔离级别

> 🔑 **核心要点**：并发访问可能导致数据不一致问题，数据库通过事务隔离级别来控制并发事务间的可见性，在一致性和性能之间寻求平衡。

## 可能存在的并发问题

### 是什么
脏读 (Dirty Read)：
定义：一个事务（T1）读取了另一个**尚未提交的事务（T2）**对数据的修改。
表现：如果事务 T2 随后因为某种原因回滚 (Rollback)，那么事务 T1 读取到的数据就是无效的、不存在的"脏"数据，因为它从未真正地被持久化到数据库中。
不可重复读 (Non-Repeatable Read)：
定义：一个事务（T1）在同一个事务中多次读取同一行数据，但在两次读取之间，另一个**已提交的事务（T2）**修改了这行数据。
表现：事务 T1 两次读取到的同一行数据的值不同，导致 T1 内部的逻辑混乱，破坏了其对数据的一致性视图。例如，T1 第一次读取某个商品库存是 100，T2 提交了减少库存的操作，T1 第二次读取发现库存变成了 99。
幻读 (Phantom Read)：
定义：一个事务（T1）在同一个事务中多次执行查询条件相同的范围查询（例如，WHERE 子句），但在两次查询之间，另一个**已提交的事务（T2）**插入了符合 T1 查询条件的新数据行。
表现：事务 T1 第一次查询得到的结果集是 N 行，第二次查询时发现结果集的行数增加了，就好像出现了"幻影"一样。幻读主要关注的是数据行的增减，而不是单行数据的修改。例如，T1 第一次查询工资大于 5000 的员工有 5 人，T2 提交了插入一个工资 6000 的新员工，T1 第二次查询发现有 6 人。
### 为什么
1. 事务的并发性和交错执行 (Interleaving)：
根本原因：数据库为了提高系统的吞吐量和资源利用率，不会让事务串行执行（即一个事务完全结束后才执行下一个）。相反，事务的操作是交错执行的。例如，事务 A 执行了几步，然后切换到事务 B 执行几步，再切换回事务 A。
问题：当事务 A 的操作依赖于事务 B 的某个中间状态，或者事务 B 在事务 A 未完成时就改变了事务 A 关注的数据，就会导致不一致。
2. 数据修改的可见性 (Visibility)：
为什么脏读会发生？ 脏读之所以发生，是因为数据库允许一个事务提前看到另一个事务尚未"定论"的修改。在事务 T2 未提交时，它所做的修改只存在于其私有的工作区或日志缓冲区中，或者虽然写入了数据页但尚未被持久化。如果 T1 能够直接读取到这些未落定的、不确定的修改，那么一旦 T2 回滚，T1 读取到的就是"垃圾"。这是由于缺乏严格的隔离机制，导致事务对其他事务修改的可见性过高。
为什么不可重复读会发生？ 不可重复读的根本在于，事务虽然能够读取到已提交的数据，但它不能锁定自己读取过的数据行。当 T1 第一次读取数据后，T2 提交了对同一行数据的修改，这些修改对 T1 是可见的（因为 T2 已提交）。T1 第二次读取时，由于没有机制阻止 T2 的修改，T1 看到的不再是自己第一次读到的数据，从而破坏了事务内部的"一致性读"。
为什么幻读会发生？ 幻读是不可重复读的一种特殊情况，但它更侧重于数据集合（范围）的变化，而不是单行数据。它的发生是因为事务 T1 对一个查询范围内的现有数据行加锁可能有效，但无法阻止另一个事务 T2 插入（或删除）符合 T1 查询条件的新数据行。T1 第一次查询时，它只看到了当时存在的数据，没有一个机制能"锁定"这个查询条件可能涵盖的"未来"的数据。当 T2 插入新行并提交后，T1 再次查询该范围，发现"无中生有"的新行，就好像幻影一样。这是因为行级锁无法有效锁定"不存在的行"或一个数据范围。
3. 缺乏足够的并发控制手段：
这些问题本质上都是由于在并发执行时，数据库未能提供足够的隔离级别来防止事务之间的相互影响。为了解决这些问题，数据库引入了**锁机制（共享锁、排他锁、间隙锁）和多版本并发控制（MVCC）**等并发控制技术，通过不同的组合来提供不同级别的事务隔离，从而阻止上述问题的发生。

## 事务的隔离级别

> 🔑 **核心要点**：事务隔离级别定义了并发事务之间的可见性和影响程度，是数据库在一致性和性能之间的权衡机制。

## 是什么

**事务隔离级别**是数据库系统为控制多个并发事务之间相互影响程度而提供的一组标准。它定义了：
- 一个事务在并发执行时，对其他事务所做修改的**可见性**
- 在事务执行过程中，数据可能受到其他并发事务**干扰的程度**

### 四种标准隔离级别

1. **读未提交 (Read Uncommitted)**
   - 最低级别
   - 一个事务可以读取到其他事务尚未提交的修改

2. **读已提交 (Read Committed)**
   - 一个事务只能读取到其他事务已经提交的修改

3. **可重复读 (Repeatable Read)**
   - 在同一个事务中，多次读取同一行或同一范围的数据，结果总是相同的

4. **串行化 (Serializable)**
   - 最高级别
   - 强制事务串行执行，完全避免所有并发问题

### 为什么

> 💡 **设计理念**：在多用户并发访问共享数据时，数据库需要在**最大化并发性**和**确保数据正确性**之间进行**权衡**。

#### 解决的核心问题

以上提到的三种并发问题（脏读、不可重复读、幻读）通过不同的隔离级别来解决。每个隔离级别都在一致性和性能之间做出了不同的权衡。

#### 各隔离级别的实现机制

##### 1. 读未提交 (Read Uncommitted)

**特点**：
- 允许**脏读**、**不可重复读**、**幻读**
- 不使用任何锁来阻塞读写
- 不使用MVCC
- 直接读取最新的（甚至未提交的）数据

**示例**：
```sql
时间  事务A(读未提交)              事务B                  数据状态
T1                               BEGIN                  balance=100
T2                               UPDATE balance=50      balance=50(未提交)
T3    SELECT balance → 返回50                          balance=50
T4                               ROLLBACK              balance=100
```

##### 2. 读已提交 (Read Committed)

**特点**：
- 避免**脏读**
- 允许**不可重复读**和**幻读**
- 使用MVCC
- 每次SELECT都生成新的ReadView

**实现机制**：
- 通过MVCC实现
- 写操作使用行级排他锁
- 读操作不加锁

**示例**：
```sql
时间  事务A(读已提交)              事务B                  数据状态
T1    BEGIN                                            balance=100
T2    SELECT balance → 返回100                         balance=100
T3                               UPDATE balance=50      balance=50
T4                               COMMIT                 balance=50
T5    SELECT balance → 返回50                          balance=50
```

##### 3. 可重复读 (Repeatable Read)

**特点**：
- 避免**脏读**、**不可重复读**
- 在InnoDB中也避免了**幻读**
- 事务开始时生成一个ReadView
- 使用间隙锁防止幻读

**实现机制**：
1. **快照读（Snapshot Read）**：
   - 普通SELECT是快照读
   - 使用事务开始时的ReadView
   - 看不到其他事务新插入的行

2. **当前读（Current Read）**：
   - UPDATE、DELETE等操作是当前读
   - 使用临键锁（Next-Key Locks）
   - 锁定记录及其间隙

##### 4. 串行化 (Serializable)

**特点**：
- 完全避免并发问题
- 性能最差
- 所有读写操作都会加锁

**实现机制**：
- 读操作加共享锁
- 写操作加排他锁
- 完全串行执行

#### 隔离级别对比

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 并发性能 | 主要实现机制 |
|---------|------|------------|------|----------|-------------|
| Read Uncommitted | ✓ | ✓ | ✓ | 最高 | 无锁 |
| Read Committed | × | ✓ | ✓ | 高 | MVCC + 行锁 |
| Repeatable Read | × | × | × | 中 | MVCC + 间隙锁 |
| Serializable | × | × | × | 低 | 完全锁定 |

> 注：✓ 表示会出现此问题，× 表示解决了此问题

---

### 总结与权衡

| 隔离级别     | 脏读   | 不可重复读 | 幻读   | 并发性能 | 主要实现机制（InnoDB）           |
| :----------- | :----- | :--------- | :----- | :------- | :------------------------------- |
| **Read Uncommitted** | 允许   | 允许       | 允许   | 最高     | 无锁，直接读最新数据             |
| **Read Committed** | 避免   | 允许       | 允许   | 较高     | 每次读生成新 ReadView，行级 X 锁 |
| **Repeatable Read** | 避免   | 避免       | **避免** | 中等     | 事务开始时生成 ReadView，**间隙锁/临键锁** |
| **Serializable** | 避免   | 避免       | 避免   | 最低     | 强制串行执行，读写均加表/行锁    |

# InnoDB 的并发控制机制

> 🔑 **核心要点**：InnoDB通过锁机制和MVCC两种方式实现并发控制。锁机制主要用于写操作的互斥，而MVCC则提供了高效的读写并发方案。

## InnoDB 的锁机制

### 是什么

行级锁：这是InnoDB的最细粒度锁，当一个事务修改某几行，通过只锁定修改的这几行来锁定被修改的表。
里面有共享锁S锁和排他锁X锁（类似Java中的读写锁）
意向锁：引入意向锁来解决行锁和表锁的兼容性问题，他是一个表级锁，目的是表示事务即将或已经对某几行加锁。里面也有意向共享锁和意向排他锁。
间隙锁：间隙锁是InnoDB在解决可重复读情境下引入的锁机制，只锁定一个范围而不是具体行。
为了解决幻读问题在可重复读RR隔离级别下引入的。确保一个事务在查询某个范围的时候，这个范围内的行数不会改变。
临键锁：默认在可重复读情况下的锁，结合了行锁和间隙所。锁定一个左开右闭的区间，是防止幻读的默认策略，保证了一个范围内的读取不仅现有行不被修改，新的行也不被修改。
记录锁：
行锁中的S锁和X锁都是记录锁，锁定具体的行。
### 为什么
实现高并发性 (Row-Level Locking)：

原因：数据库的瓶颈往往在于并发操作时的资源争抢。如果每次操作都锁定整个表，那么所有并发事务都必须串行执行，系统吞吐量极低。

目标：通过行级锁，InnoDB 允许不同的事务同时修改表中不同的行，最大化了并发度。当事务 A 修改第 1 行时，事务 B 可以修改第 2 行，互不影响。这显著提升了 OLTP (在线事务处理) 系统的并发能力。

保证事务隔离性 (Isolation Levels)：

原因：数据库通过事务隔离级别（如读已提交、可重复读）来控制并发事务之间的可见性。不同的隔离级别需要不同的锁策略来防止并发问题。

目标：

脏读：通过行级 X 锁（在事务未提交前持有），防止其他事务读取到未提交的数据。

不可重复读：通过行级 X 锁或 S 锁（结合 MVCC），确保一个事务在多次读取同一行时看到的数据是稳定的。

幻读：这是最难解决的并发问题。InnoDB 默认在 REPEATABLE READ 隔离级别下，通过引入间隙锁和临键锁**，来锁定数据范围，从而阻止其他事务在该范围内插入新行，从根本上防止了幻读。虽然这会降低一点并发性，但保证了该隔离级别下的数据严格一致性。

解决锁升级问题 (Intention Locks)：

原因：如果没有意向锁，当一个事务想对整个表进行表级操作（如 LOCK TABLES 或 ALTER TABLE）时，数据库需要扫描整个表，检查每一行是否有被其他事务加了行级锁。这会带来巨大的性能开销。

目标：意向锁作为一种表级信号，它告诉数据库："这张表里有（或将有）行级锁存在"。这样，当另一个事务尝试获取表级锁时，只需要检查表上的意向锁即可，无需扫描所有行，避免了锁检查的性能开销，提高了表级操作的效率。

死锁检测与恢复：

原因：当多个事务相互等待对方释放资源（锁）时，就会发生死锁。

目标：InnoDB 内置了死锁检测机制。当检测到死锁时，它会选择一个成本最低的事务进行回滚 (Rollback)，释放其持有的锁，从而打破死锁，让其他事务继续执行。这确保了系统的可用性，避免了无限期阻塞。

支持 MVCC (Multi-Version Concurrency Control)：

原因：纯粹的锁机制在读写并发高时，读事务也会被写事务阻塞。

目标：InnoDB 结合了锁和 MVCC。对于读操作，在 READ COMMITTED 和 REPEATABLE READ 隔离级别下，通常不会加锁，而是通过 Undo Log 读取数据的历史版本快照。这使得读操作几乎不阻塞写操作，写操作也不阻塞读操作，极大地提升了读写并发性能。只有在需要读取最新数据或加锁读时（如 SELECT ... FOR UPDATE），才会显式加锁。

### 间隙锁 (Gap Lock) 的体现与场景

**是什么：** 间隙锁是 InnoDB 在可重复读 (Repeatable Read, RR) 隔离级别下为了防止**幻读**而引入的一种锁机制。它锁定的是**索引记录之间的间隙**，或者**第一条索引记录之前的间隙**，以及**最后一条索引记录之后的间隙**。它不锁定实际的记录，而是锁定**不存在的记录范围**。

**加锁范围与体现：**
* **锁定间隙：** 当你执行一个范围查询，并且这个范围内有数据，或者没有数据，间隙锁都会锁定这个范围。例如，索引列 `id` 有值 10, 20, 30。
    * `SELECT * FROM t WHERE id > 15 AND id < 25 FOR UPDATE;` 此时，间隙锁会锁定 `(10, 20)` 和 `(20, 30)` 之间的潜在插入位置。
    * `SELECT * FROM t WHERE id > 30 FOR UPDATE;` 间隙锁会锁定 `(30, +∞)` 这个范围。
    * `SELECT * FROM t WHERE id < 10 FOR UPDATE;` 间隙锁会锁定 `(-∞, 10)` 这个范围。
* **防止插入：** 间隙锁的主要作用是防止其他事务在被锁定的间隙内**插入新的数据行**。如果在上述例子中，事务A持有了 `(10, 20)` 的间隙锁，那么其他事务就不能插入 `id=15` 的记录。
* **与行锁独立：** 间隙锁只影响插入操作，它不阻止其他事务对间隙内已有记录的修改或删除（除非有临键锁）。

**适用场景：**
* **RR 隔离级别下的范围更新/删除：** 当在 RR 级别下对一个范围进行 `UPDATE ... WHERE ...` 或 `DELETE ... WHERE ...` 操作时，InnoDB 会自动加上间隙锁，以确保在事务提交前，没有新的记录能插入到这个范围内，从而防止幻读。
* **显式加锁的范围查询：** `SELECT ... FOR UPDATE` 或 `SELECT ... LOCK IN SHARE MODE` 在 RR 级别下进行范围查询时，也会加上间隙锁。

---

### 临键锁 (Next-Key Lock) 的体现与场景

**是什么：** 临键锁是 InnoDB 在 RR 隔离级别下**默认的行锁方式**，它是**行锁和间隙锁的组合**。它锁定的是一个**左开右闭的区间 `(A, B]`**，其中 A 是上一个索引记录，B 是当前索引记录。

**加锁范围与体现：**
* **锁定记录本身和其之前的间隙：** 临键锁会锁定它所覆盖的**当前索引记录**以及该记录**之前**到上一条索引记录之间的**间隙**。
* **防止修改和插入：** 临键锁既能阻止其他事务修改或删除当前记录，也能阻止其他事务在被锁定的间隙内插入新记录。

**加锁规则（简单说明）：**
当查询一条记录时，InnoDB 通常会锁定扫描到的索引项对应的临键区间。
* **等值查询且唯一索引命中：** 如果查询条件是等值匹配，并且使用了唯一索引（包括主键），则只会加**记录锁 (Record Lock)**，不加间隙锁，因为唯一索引可以保证没有幻读。
* **非唯一索引或范围查询：** 如果是非唯一索引或者范围查询，为了防止幻读，InnoDB 就会使用临键锁。

**示例（基于索引列 `id`，值为 10, 20, 30）：**
* `SELECT * FROM t WHERE id = 20 FOR UPDATE;` (假设 `id` 是非唯一索引)
    * 这会加一个临键锁 `(10, 20]`，锁定记录 20 和其前面的间隙 `(10, 20)`。其他事务不能修改 20，也不能插入 15。
* `SELECT * FROM t WHERE id > 10 AND id <= 20 FOR UPDATE;`
    * 可能会对记录 20 及其之前的间隙加临键锁 `(10, 20]`。
* `SELECT * FROM t WHERE id > 30 FOR UPDATE;`
    * 对 `(30, +∞)` 加间隙锁。
* `SELECT * FROM t WHERE id < 10 FOR UPDATE;`
    * 对 `(-∞, 10)` 加间隙锁。

**适用场景：**
* **InnoDB RR 隔离级别下的默认锁行为：** 几乎所有在 RR 隔离级别下的 `UPDATE`、`DELETE` 和 `SELECT ... FOR UPDATE` 操作，在遍历索引时都会使用临键锁，以同时防止不可重复读和幻读。
* **保证范围查询的一致性：** 在一个事务中，如果你执行一个范围查询，并希望确保在这个事务结束前，这个范围内的数据（包括新的插入）保持不变，临键锁是关键。

### 死锁检测的原理是什么？

**是什么：** 死锁是指两个或多个事务在执行过程中，因争夺资源而造成的一种互相等待的现象，若无外力作用，它们都将无法继续执行。InnoDB 存储引擎提供了一种**死锁检测机制**，当检测到死锁发生时，会选择一个"牺牲者"事务（通常是成本最小的事务）进行回滚，从而打破死锁，让其他事务继续执行。

**原理：**
1.  **等待图 (Wait-for Graph) 构建：** InnoDB 在内部维护一个**有向图**，用于表示事务之间的锁等待关系。
    * **节点 (Node)：** 图中的节点代表事务。
    * **边 (Edge)：** 如果事务 A 正在等待事务 B 释放锁资源，那么就从 A 到 B 画一条有向边。
2.  **周期检测：** 死锁检测器会周期性地检查这个等待图。如果图中存在**环路 (Cycle)**，就说明发生了死锁。
    * 例如：事务 A 等待事务 B，事务 B 等待事务 C，事务 C 等待事务 A。形成一个 A -> B -> C -> A 的环，即为死锁。
3.  **回滚"牺牲者"：** 一旦检测到环路，InnoDB 会启动死锁回滚机制。它会评估环路中涉及的所有事务，选择一个**回滚成本最小**的事务（例如，修改行数最少，或者执行时间最短的事务）作为牺牲者，将其回滚。被回滚的事务会释放它所持有的所有锁，从而打破死锁循环，使其他事务得以继续执行。

**优点：** 自动检测和处理死锁，减少了DBA的人工干预，提高了系统的可用性。
**缺点：** 频繁进行死锁检测会消耗一定的CPU资源，在高并发场景下可能会有性能开销。

---

### 如何避免死锁？

避免死锁是一个更积极的策略，比死锁检测更加重要，因为它能从根本上减少死锁的发生。

1.  **以固定的顺序访问资源：** * 如果所有事务都按照**相同的顺序**获取锁（例如，总是先锁定表 A 的行，再锁定表 B 的行），那么死锁发生的概率会大大降低。这是避免死锁最有效的方法之一。
    * **示例：** 事务 A (获取锁A -> 获取锁B)，事务 B (获取锁A -> 获取锁B)。如果两个事务都想同时获取A和B，它们会都在获取A时竞争，谁先拿到谁就继续，不会出现互相等待。
2.  **批量操作，减少锁的持有时间：** * 尽可能地减少事务持有锁的时间。将需要锁定的操作集中起来，一次性完成，而不是分散在事务的各个阶段。
    * **示例：** 避免在事务中等待用户输入、网络请求等耗时操作。
3.  **大事务拆小事务：** * 将一个复杂的、涉及大量数据操作的事务拆分成多个小的事务。这样可以减少单个事务持有锁的时间和资源范围。
4.  **为查询增加合适的索引：** * 不合适的索引会导致扫描的行数过多，从而可能锁定不必要的行，增加死锁的风险。优化器在选择执行计划时，如果能通过索引精准定位，就能减少加锁范围。
5.  **减少不必要的锁竞争：** * 在并发度非常高的场景，考虑使用**乐观锁**（例如，通过版本号字段）来替代部分悲观锁，减少锁等待。
    * 考虑使用更细粒度的锁，但 InnoDB 默认就是行级锁，这方面更多是优化 SQL。
6.  **降低隔离级别（慎用）：** * 虽然可以通过降低隔离级别（如从 RR 降到 RC）来减少锁的持有，从而降低死锁概率，但这会带来其他并发问题（如不可重复读、幻读）。除非对数据一致性要求不高，否则不建议轻易调整。
7.  **避免在数据库层面执行耗时操作：** * 避免在事务中执行文件操作、网络请求等与数据库无关且耗时的操作。

## MVCC (多版本并发控制)

### 是什么

一种读不加锁写不阻塞的方案，通过给每条记录打上版本号或者时间戳的方式进行多版本管理。
当一个数据被修改的时候，他不直接用新数据修改旧数据，而是通过在undolog中储存旧版本的数据
然后创建一个新的版本，新的版本和旧的版本通过
每个新的版本会包含 事务ID,回滚指针，行ID。
RC级别下每次都会创建一个新的Read View
读取最新提交记录，如果最新版本的事务ID大于当前ID,那么他会找到一个合适的版本进行读取。
RR级别下只有事务开始的时候会创建一个ReadView 这样可以保证在整个事务期间读取到的都是同一个版本。
### 为什么
提升了并发读取性能，这样可以让读操作不加锁，其实就是类似一个快照的形式。
减少锁的竞争。
可以实现事务隔离级别。
提供了更高的并发性，读操作不会阻塞写操作，写操作也不会阻塞读操作。

---

### ReadView 的具体判断逻辑（可见性算法）

**是什么：** ReadView 是 MVCC（多版本并发控制）实现的关键组成部分，它是在**事务启动时生成的一个内存对象**，用于判断当前事务能够"看到"哪个版本的数据。本质上，它定义了当前事务的一个**一致性快照 (Consistent Snapshot)**。

**ReadView 包含的核心信息：**
* `m_ids` (trx_ids)：一个活跃事务ID列表，记录了在生成 ReadView 时，当前系统中**所有活跃的（即未提交的）读写事务的 ID 列表**。
* `min_trx_id` (up_limit_id)：`m_ids` 列表中**最小的事务 ID**。比这个 ID 小的事务（如果已提交）都是可见的。
* `max_trx_id` (low_limit_id)：在生成 ReadView 时，系统**下一个将被分配的事务 ID**（即当前已分配的最大事务 ID + 1）。比这个 ID 大的事务都是不可见的。
* `creator_trx_id`：创建这个 ReadView 的事务本身的 ID。

**可见性判断算法：**
当一个事务尝试读取一行数据时，会根据该行的 `trx_id`（最后修改该行的事务 ID）和当前的 ReadView 进行比较，判断该行数据的某个版本是否对当前事务可见：

1.  **情况一：当前行 `trx_id < min_trx_id`**
    * 如果行的 `trx_id` **小于** ReadView 中 `min_trx_id`，表示修改该行的事务在当前 ReadView 生成之前就已经**提交**了。
    * **结论：该行数据版本**对当前事务**可见**。

2.  **情况二：当前行 `trx_id >= max_trx_id`**
    * 如果行的 `trx_id` **大于或等于** ReadView 中 `max_trx_id`，表示修改该行的事务是在当前 ReadView 生成之后才启动的。
    * **结论：该行数据版本**对当前事务**不可见**。

3.  **情况三：`min_trx_id <= 当前行 trx_id < max_trx_id`**
    * 这种情况表示修改该行的事务是在当前 ReadView 生成时"活跃"的事务，或者是在 ReadView 生成后才启动的事务（但ID小于max_trx_id）。此时需要进一步判断：
    * **情况 3.1：`trx_id == creator_trx_id`**
        * 如果行的 `trx_id` 等于创建 ReadView 的事务本身的 ID，说明是当前事务自己修改的。
        * **结论：该行数据版本**对当前事务**可见**。
    * **情况 3.2：`trx_id` 在 `m_ids` 列表中**
        * 如果行的 `trx_id` **存在于** `m_ids` 列表中，表示修改该行的事务在 ReadView 生成时是活跃的，并且尚未提交。
        * **结论：该行数据版本**对当前事务**不可见**。需要沿着 Undo Log 版本链找上一个可见版本。
    * **情况 3.3：`trx_id` 不在 `m_ids` 列表中**
        * 如果行的 `trx_id` **不在** `m_ids` 列表中，表示修改该行的事务在 ReadView 生成时已经**提交**了。
        * **结论：该行数据版本**对当前事务**可见**。

如果当前版本不可见，就会沿着 Undo Log 的版本链向前追溯，直到找到一个对当前事务可见的版本。

### 结合 Undo Log 版本链，更清晰地解释 MVCC 工作原理

### MVCC (多版本并发控制) 的工作原理

MVCC 是通过**保存数据在某个时间点的快照**来实现的，它使得读操作可以不阻塞写操作，写操作也不阻塞读操作，极大地提升了数据库的并发性能。它的核心是**版本链**和 **ReadView**。

1.  **数据行的隐藏列：**
    InnoDB 为每行数据添加了几个隐藏列：
    * `DB_TRX_ID`：最近一次修改（插入或更新）该行的事务 ID。
    * `DB_ROLL_PTR`：回滚指针，指向这条记录在上一个版本在 Undo Log 中的位置。
    * `DB_ROW_ID`：隐藏的行 ID，如果表没有主键且没有唯一非空索引，InnoDB 会自动生成。

2.  **Undo Log 版本链 (Version Chain)：**
    * 每当对一条记录进行 `UPDATE` 或 `DELETE` 操作时，InnoDB 并不会直接覆盖原始数据，而是：
        1.  将**旧版本的数据**写入到 **Undo Log** 中。
        2.  将旧版本的 `DB_TRX_ID` 和 `DB_ROLL_PTR` 更新，指向 Undo Log 中的上一个版本。
        3.  更新当前行的 `DB_TRX_ID` 为**当前事务的 ID**。
    * 这样，每次对记录的修改都会在 Undo Log 中创建一个历史版本，并且通过 `DB_ROLL_PTR` 形成一个**版本链**，链条的**头部是最新的数据，尾部是最老的快照**。
    * **示例：**
        * 初始数据：`id=1, name='A'`, `DB_TRX_ID=T1`
        * 事务 T2 `UPDATE name='B' WHERE id=1;`
            * Undo Log 记录旧版本 `(id=1, name='A', DB_TRX_ID=T1, DB_ROLL_PTR=NULL)`
            * 当前数据变为 `id=1, name='B'`, `DB_TRX_ID=T2`, `DB_ROLL_PTR` 指向 Undo Log 中的旧版本
        * 事务 T3 `UPDATE name='C' WHERE id=1;`
            * Undo Log 记录旧版本 `(id=1, name='B', DB_TRX_ID=T2, DB_ROLL_PTR=指向T1版本)`
            * 当前数据变为 `id=1, name='C'`, `DB_TRX_ID=T3`, `DB_ROLL_PTR` 指向 Undo Log 中的 T2 版本
        * 这就形成了一个 `C (T3) -> B (T2) -> A (T1)` 的版本链。

3.  **ReadView 与版本链的协同：**
    * 当一个事务执行**普通 SELECT (快照读)**时，它会生成一个 **ReadView**。
    * 然后，它会从当前数据行的**最新版本**开始，根据**上述 ReadView 可见性算法**判断这个版本是否可见：
        * 如果可见，就直接返回这个版本的数据。
        * 如果不可见（例如，该版本是由一个还未提交的事务修改的），就沿着 `DB_ROLL_PTR` 指针**回溯到 Undo Log 中的上一个历史版本**。
        * 重复这个判断过程，直到找到一个对当前 ReadView **可见**的历史版本，或者版本链已经到头。
    * 这个过程确保了在不同隔离级别下，事务读取到的都是一个**一致性快照**的数据，避免了脏读和不可重复读。

**隔离级别与 ReadView 的关系：**
* **读已提交 (RC)：** **每次 `SELECT` 语句执行时都会重新生成一个 ReadView**。这意味着在一个事务中，如果其他事务提交了修改，下一次 `SELECT` 就能看到最新的已提交数据，因此可能出现不可重复读。
* **可重复读 (RR)：** **事务开始时只生成一个 ReadView，并在整个事务生命周期内都使用这个 ReadView**。这意味着在事务期间，无论其他事务如何提交修改，本事务看到的都是启动时的数据快照，从而避免了不可重复读和（结合间隙锁）幻读。

# 事务
### 是什么
是一个数据库操作的逻辑单元，它由一系列的数据库操作组成，这些操作要么全部成功，要么全部失败。事务的四个特性被称为ACID特性：
原子性 (Atomicity)：事务中的所有操作要么全部执行成功，要么全部撤销。即使在系统崩溃或错误发生时，也能保证事务的完整性。  
一致性 (Consistency)：事务执行前后，数据库的状态必须保持一致。即事务的执行不会破坏数据库的完整性约束。
隔离性 (Isolation)：多个事务并发执行时，每个事务都应该像是
独立执行的，互不干扰。即使多个事务同时操作同一数据，也不会影响各自的结果。
持久性 (Durability)：一旦事务提交，其结果是永久性的，即使系统
崩溃或重启，已提交的事务结果也不会丢失。
### 为什么
可以防止部分更新，有些操作比如说扣款和扣减库存，这两个操作要么都成功，要么都失败，不能出现扣款成功但库存没有减少的情况。
可以防止并发访问下的数据不一致问题，比如说两个用户同时购买同一件商品，可能会导致库存超卖。
可以保证数据的完整性和一致性，确保数据库在任何时候都处于一个

# 事务的隔离级别

> 🔑 **核心要点**：事务隔离级别定义了并发事务之间的可见性和影响程度，是数据库在一致性和性能之间的权衡机制。

## 是什么

**事务隔离级别**是数据库系统为控制多个并发事务之间相互影响程度而提供的一组标准。它定义了：
- 一个事务在并发执行时，对其他事务所做修改的**可见性**
- 在事务执行过程中，数据可能受到其他并发事务**干扰的程度**

### 四种标准隔离级别

1. **读未提交 (Read Uncommitted)**
   - 最低级别
   - 一个事务可以读取到其他事务尚未提交的修改

2. **读已提交 (Read Committed)**
   - 一个事务只能读取到其他事务已经提交的修改

3. **可重复读 (Repeatable Read)**
   - 在同一个事务中，多次读取同一行或同一范围的数据，结果总是相同的

4. **串行化 (Serializable)**
   - 最高级别
   - 强制事务串行执行，完全避免所有并发问题

### 为什么

> 💡 **设计理念**：在多用户并发访问共享数据时，数据库需要在**最大化并发性**和**确保数据正确性**之间进行**权衡**。

#### 解决的核心问题

1. **脏读 (Dirty Read)**
   - 定义：一个事务读取了另一个**未提交事务**修改的数据
   - 风险：如果未提交事务最终回滚，则读取到的数据是无效的

2. **不可重复读 (Non-Repeatable Read)**
   - 定义：同一事务中，两次读取同一行数据，但数据被**另一个已提交事务修改**
   - 影响：导致两次读取结果不一致

3. **幻读 (Phantom Read)**
   - 定义：同一事务中，两次执行相同的**范围查询**，但第二次查询时，由于**另一个已提交事务插入/删除了数据**
   - 结果：导致结果集中的行数不同

#### 各隔离级别的实现机制

##### 1. 读未提交 (Read Uncommitted)

**特点**：
- 允许**脏读**、**不可重复读**、**幻读**
- 不使用任何锁来阻塞读写
- 不使用MVCC
- 直接读取最新的（甚至未提交的）数据

**示例**：
```sql
时间  事务A(读未提交)              事务B                  数据状态
T1                               BEGIN                  balance=100
T2                               UPDATE balance=50      balance=50(未提交)
T3    SELECT balance → 返回50                          balance=50
T4                               ROLLBACK              balance=100
```

##### 2. 读已提交 (Read Committed)

**特点**：
- 避免**脏读**
- 允许**不可重复读**和**幻读**
- 使用MVCC
- 每次SELECT都生成新的ReadView

**实现机制**：
- 通过MVCC实现
- 写操作使用行级排他锁
- 读操作不加锁

**示例**：
```sql
时间  事务A(读已提交)              事务B                  数据状态
T1    BEGIN                                            balance=100
T2    SELECT balance → 返回100                         balance=100
T3                               UPDATE balance=50      balance=50
T4                               COMMIT                 balance=50
T5    SELECT balance → 返回50                          balance=50
```

##### 3. 可重复读 (Repeatable Read)

**特点**：
- 避免**脏读**、**不可重复读**
- 在InnoDB中也避免了**幻读**
- 事务开始时生成一个ReadView
- 使用间隙锁防止幻读

**实现机制**：
1. **快照读（Snapshot Read）**：
   - 普通SELECT是快照读
   - 使用事务开始时的ReadView
   - 看不到其他事务新插入的行

2. **当前读（Current Read）**：
   - UPDATE、DELETE等操作是当前读
   - 使用临键锁（Next-Key Locks）
   - 锁定记录及其间隙

##### 4. 串行化 (Serializable)

**特点**：
- 完全避免并发问题
- 性能最差
- 所有读写操作都会加锁

**实现机制**：
- 读操作加共享锁
- 写操作加排他锁
- 完全串行执行

#### 隔离级别对比

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 并发性能 | 主要实现机制 |
|---------|------|------------|------|----------|-------------|
| Read Uncommitted | ✓ | ✓ | ✓ | 最高 | 无锁 |
| Read Committed | × | ✓ | ✓ | 高 | MVCC + 行锁 |
| Repeatable Read | × | × | × | 中 | MVCC + 间隙锁 |
| Serializable | × | × | × | 低 | 完全锁定 |

> 注：✓ 表示会出现此问题，× 表示解决了此问题
