### **. 概述：数据表示 (Overview: Representations)**

数据库系统需要将数据持久化存储在磁盘上。数据在磁盘上的组织方式是数据库管理系统（DBMS）的基础。数据通常被组织成**文件（Files）**，文件由一系列的**页面（Pages）**组成，而每个页面又包含一系列的**记录（Records）**。

例如，一个员工表的数据可以表示为多条记录，每条记录包含员工的SSN、姓氏、名字、年龄和薪水等字段。

**为什么要有Page？**
直接操作磁盘上的字节效率非常低。Page是DBMS和操作系统之间交互的基本单位。DBMS不会一次只读写一个字节或一条记录，而是以Page为单位进行读写，这大大提高了I/O效率。

**数据管理层级:**
* **磁盘空间管理器 (Disk Space Manager):** 负责在物理磁盘/文件中读写页面。它处理的是物理存储细节，比如哪个页面在磁盘的哪个扇区。
* **缓冲区管理器 (Buffer Manager):** 负责在内存中管理页面（缓存）。当DBMS需要某个页面时，它会先问缓冲区管理器，如果页面已经在内存中，就直接提供；如果不在，缓冲区管理器会调用磁盘空间管理器将其从磁盘加载到内存。所有的数据库操作（如查询、更新）都只在内存中的页面上进行。

### **数据库文件结构 (DB File Structures)**

一个数据库文件（DB FILE）是页面的集合，每个页面又包含记录的集合。

DBMS向更高层（比如SQL查询处理器）提供了一系列API来操作这些文件:
* 插入、删除、修改记录。
* 通过记录ID（Record ID）获取特定记录。记录ID是一个指针，编码了页ID（pageID）和页面上的位置（location on page）。例如，`Record ID = (Page 2, Record 4)` 可能表示在文件中的第2个页面上的第4条记录。
* 扫描所有记录，可能带有某些检索条件。

数据库文件可以跨越多个操作系统文件，甚至直接在“裸”磁盘设备上运行。

**文件组织方式:**

* **无序堆文件 (Unordered Heap Files):**
    * 记录在页面中随意放置，没有特定的顺序。
    * 随着文件增长或收缩，页面会被分配或释放。
    * 为了支持记录级别的操作，需要跟踪文件中的页面、页面上的空闲空间以及页面上的记录。
    * **作为链表实现的堆文件 (Heap File Implemented as List):** 早期简单的方法可能是在每个页面头部存储指向下一个和上一个页面的“指针”（Page ID）。这种方式的问题在于，要找到一个有足够空间插入新记录的页面，需要从头开始遍历链表，效率低下。
    * **更好的方法：使用页面目录 (Better: Use a Page Directory):** DBMS维护一个或多个特殊的**头页面（Header Page）**，这些头页面组成一个“页面目录”。页面目录中的每个条目都记录了它所引用的数据页面的信息，特别是该数据页的**空闲字节数（#free bytes on the referenced page）**。当需要插入新记录时，DBMS只需查看头页面（通常会在内存缓存中，访问速度快），就能快速找到一个有足够空闲空间的页面。

* **聚簇堆文件 (Clustered Heap Files):** 记录和页面是分组的。
* **排序文件 (Sorted Files):** 页面和记录都按照排序顺序排列。
* **索引文件 (Index Files):**
    * 例如B+树、线性哈希等。索引文件可能直接包含记录，或者指向其他文件中的记录。
    * 索引是用于高效地基于值进行查询的文件结构，例如查找特定部门的学生或满足特定GPA和头发颜色的学生。

### **页面布局 (Page Layout)**

页面布局决定了记录如何在页面内组织。

**页面基础：页头 (Page Basics: The Header):**
* 页头可能包含：记录数量、空闲空间、指向下一个/上一个页面的指针、位图（Bitmaps）、槽表（Slot Table）等信息。
* **需要解决的问题 (Things to Address):** 记录长度是固定还是可变的？如何通过记录ID查找记录？如何添加和删除记录？

**页面布局选项取决于 (Options for Page Layouts Depends on):**
* 记录长度（固定或可变）。
* 页面填充方式（紧密填充或非紧密填充）。

**A. 固定长度记录，紧密填充 (Fixed Length Records, Packed):**
* 记录被紧密地打包在一起。
* 记录ID通常是`(pageId, record number in page)`，因为偏移量可以计算出来。
* 添加记录很简单，只需追加到末尾。
* **删除记录的问题 (Delete?):** 如果删除中间的记录，为了保持紧密填充，**必须重新排列后面的记录来填补空隙**。这会导致记录ID（如果它们是指向页面内序号的话）发生变化，如果这些记录ID在其他文件中被引用（例如索引），那么更新它们的代价会很高。

**B. 固定长度记录，非紧密填充 (Fixed Length Records: Unpacked):**
* 使用位图（Bitmap）来表示哪些“槽位”包含记录。
* 记录ID是页面内的记录编号。
* 插入记录：查找第一个空槽位。
* 删除记录：清除位图中的相应位。
* **优点:** 删除记录时，不需要移动其他记录，因此记录ID保持稳定，外部引用（如索引）无需更新。

**C. 可变长度记录的页面布局：槽页 (Variable Length Records: Slotted Page):**
处理可变长度记录的挑战在于如何知道每条记录的起始位置，以及在添加和删除记录时如何管理空间。槽页是解决这些问题的常用且优雅的方案。

* **元数据后置到页脚 (Relocate metadata to footer):**
    * 在槽页设计中，页脚通常包含**槽目录（Slot Directory）**。
    * **为什么将元数据放在页脚而不是页头？** 这是为了**最大化页面空间利用率**和**避免频繁的页面内部数据移动**。
        * 在槽页中，**实际的记录数据通常从页面的开头（低地址）向高地址方向增长**。
        * 而**槽目录（元数据）则从页面的末尾（高地址，即页脚）向低地址方向增长**。
        * 这种“两端向中间增长”的设计，允许页面最大限度地容纳可变长度的记录，直到记录和槽目录“相遇”时，页面就满了。
        * 此外，当插入或删除记录时，即使记录数据在页面内是分散的，只要槽目录的指针正确指向它们，就不需要移动实际数据。只有在碎片化严重需要**页面重组**时，才会移动数据，但此时槽目录的指针会相应更新，记录ID保持不变。
    * **关于“遍历”的问题：** 槽目录不是用来遍历记录数据的，而是一个**查找表**。当通过记录ID `(PageID, SlotNumber)` 查找记录时，DBMS会直接通过 `SlotNumber`（作为索引）在槽目录中找到对应的槽位，并获取记录的起始偏移量和长度，从而实现**直接访问**，而非遍历。

* **槽目录 (Slot Directory):**
    * 槽目录包含指向页面内空闲空间的指针，以及每个记录的长度和指向记录起始位置的指针。这些指针通常以反向顺序存储（从右侧开始增长）。
    * 记录ID通常是槽目录中的位置（例如，页面上的第4条记录）。

* **删除记录 (Delete Record):**
    * 删除记录时，只需将槽目录中对应槽位的指针设为null。这不会影响其他记录的指针。

* **插入记录 (Insert Record):**
    * 将新记录放置在页面上的空闲空间中。
    * 在槽目录中下一个可用的槽位创建指针/长度对。
    * 更新空闲空间指针。

* **碎片化 (Fragmentation):**
    * 由于记录的删除和插入，页面内可能会出现碎片化的空闲空间。
    * **重组页面内数据 (Reorganize data on page):** 当碎片化严重时，可以重新组织页面内的数据，将记录紧密排列，回收碎片。这种重组是安全的，因为记录ID（即槽目录中的位置）不会改变。
    * 何时重组？可以在删除时立即重组，或者等到碎片阻碍新记录添加时再重组。

* **槽位增长 (Growing Slots):**
    * 槽目录的大小是可变的，会根据页面上记录的数量而增长或收缩。
    * 槽位从页面的末尾向内增长，而记录从页面的开头向内增长。

**槽页总结 (Slotted Page: Summary):**
* 槽页是处理可变长度和固定长度记录的常用且有效的方法。对于固定长度记录也很好用。

### **记录格式 (Record Formats)**

**为什么需要不同的记录格式？**
数据库需要高效地存储各种类型的数据，同时也要能够快速地访问每个字段。

**核心目标:**
* 记录在内存和磁盘上都要**紧凑**，以节省存储空间并减少I/O。
* 能够**快速访问字段**，因为这是查询处理中最频繁的操作之一。

**A. 固定长度字段 (Fixed Length Fields):**
* **特点:** 每个字段的长度都是固定的。
* **优点:** 访问第 `i` 个字段非常快，通过简单的算术运算即可直接计算出偏移量。
* **缺点:** 如果存在大量空值（NULL），或者使用了`CHAR(N)`但实际存储的字符串很短，就会造成空间浪费。

**B. 可变长度字段 (Variable Length Fields):**

* **填充 (Padding):** 可以使用填充来使可变长度字段也具有固定长度。但这会造成空间浪费，并且如果字段内容过长，可能无法容纳。
* **使用分隔符 (Delimiters, i.e., CSV):** 字段之间用特殊字符（如逗号）分隔。
    * **缺点:** 需要扫描整个记录才能访问特定字段。如果文本内容本身包含分隔符，会导致解析错误。
* **在字段前存储长度信息 (Store length information before fields):** 在每个可变长度字段前存储其长度。
    * **缺点:** 仍然需要扫描来访问特定字段。
* **最佳实践：引入记录头 (Introduce a record header):**
    * 这是最常见且灵活的方案。在每条记录的开头添加一个小的**记录头（Record Header）**。
    * **记录头通常包含:**
        * **空值位图 (Null Bitmap):** 一个位图，每个位对应一个字段，指示该字段是否为NULL。
        * **偏移量数组 (Offset Array):** 对于变长字段，记录头中包含一个偏移量数组。数组的每个条目存储对应字段在记录内部的起始偏移量。
    * **偏移量数组相比直接记录长度的优势：**
        * **直接访问字段 (Direct Access to Fields):** 这是最主要的优势。如果要访问第 `i` 个字段，DBMS不需要扫描前面的字段。它只需要读取记录头中的**偏移量数组**，然后直接从数组中获取 `Offset_Field_i`。通过这个偏移量，DBMS可以直接跳到记录数据区域中第 `i` 个字段的起始位置。这大大提高了访问特定字段的效率，特别是对于有大量字段或前面字段很长的记录。
        * **易于处理空值 (Easier Null Handling):** 记录头中的空值位图可以非常高效地标记哪些字段是NULL，无需为NULL字段存储实际数据。

### **系统目录 (System Catalogs)**

系统目录（System Catalogs）就像数据库自己的“元数据图书馆”。它存储了关于数据库中所有对象（表、索引、视图等）的定义和属性信息。

**系统目录本身也是作为关系（表）存储的** 这意味着DBMS使用它自己的表来管理关于其他表的信息。

**系统目录中存储的信息 (Information stored in System Catalogs):**
* **对于每个关系（表）:** 名称、文件位置、文件结构（例如，堆文件）。每个属性的名称和类型。每个索引的名称。完整性约束。
* **对于每个索引:** 结构（例如，B+树）和搜索键字段。
* **对于每个视图:** 视图名称和定义。
* 此外，还包括统计信息、授权信息、缓冲区池大小等。

例如，PostgreSQL的 `information_schema.columns` 和 SQLite的 `sqlite_master` 都展示了数据库如何以表的形式存储其自身的元数据。

### **页面大小选择的考量 (Page Size Considerations)**

数据库页面通常采用 4KB、8KB 或 16KB 等大小，这与操作系统底层页面管理和磁盘I/O效率有着密切的关系。

* **与操作系统页面大小的匹配 (Operating System Page Size Matching):** 现代操作系统进行内存管理时，也是以固定大小的“页”（或“内存页”）为单位的，最常见的大小是 **4KB**。如果数据库的页面大小与操作系统的内存页大小一致或为其倍数，可以减少不必要的内存拷贝和页面碎片，提高I/O效率。当一个数据库页面被加载到内存时，它可以恰好占据操作系统的整数个内存页，简化了内存管理和地址映射，并有助于**减少TLB（转换后备缓冲区）未命中**，从而加速内存访问。

* **磁盘I/O的效率 (Disk I/O Efficiency):** 磁盘I/O是数据库系统中最慢的操作之一。为了最大化吞吐量并减少寻道时间的影响，DBMS会尝试一次性读取相对较大的数据块。一个太小的页面会导致频繁的I/O操作，每次I/O的固定开销会降低效率。而一个太大的页面则可能导致过度读取不必要的数据，并增加内存消耗。选择这些大小是为了在单次I/O操作能传输更多数据和减少不必要的数据传输之间找到一个平衡点，并且这些大小通常是磁盘物理块大小的倍数。

* **CPU缓存与总线带宽 (CPU Cache and Bus Bandwidth):** 页面大小的设计也会考虑与CPU缓存行大小的匹配，以减少缓存未命中。同时，合适的页面大小可以更好地利用内存总线的带宽。

* **数据结构的效率 (Data Structure Efficiency):** 页面是B+树等索引结构的基本单位。合适的页面大小可以决定B+树的扇出（fan-out）因子，通常更宽的B+树意味着更少的磁盘I/O来遍历索引，从而提高查询性能。
