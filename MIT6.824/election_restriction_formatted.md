#### 5.3 选举限制机制

Raft使用投票的方式来阻止候选人实现选举，除非这个候选人包含了所有已提交的日志。

##### 限制原理

- 候选人为了赢得选举必须联系大部分节点
- 这意味着每个已经提交的日志条目在这些服务器节点中肯定存在于某一个节点之上
- 如果候选人日志至少和大多数服务器节点一样新，那么他一定持有了所有已经提交的条目

##### 问题分析：为什么需要选举限制？

这个规则要解决的核心问题是：**防止一个已经被大多数节点确认、但尚未"正式宣布"的日志，被后续更高任期的 Leader 给覆盖掉。**

##### 违反规则的灾难性后果示例

让我们通过一个具体例子来看，如果不遵守选举限制规则会发生什么。

**场景设定：**
- 5台服务器：S1, S2, S3, S4, S5
- "过半数"指的是3台或以上服务器
- 假设有一个错误的Raft实现，允许Leader仅凭"数量过半"就提交之前任期的日志
- 日志记录格式为 `[任期号, 数据]`

**灾难时间线：**

**第1步：任期2**
1. S1当选为任期2的Leader
2. 客户端请求写入数据"A"
3. S1在自己的日志中追加`[T2, "A"]`，发送给S2和S3
4. S2和S3都成功接收并写入了这条日志
5. **关键点：** S1突然崩溃！还没来得及告诉S2和S3"这条日志已经可以提交了"

```
当前状态:
S1: [T2, "A"] (已崩溃)
S2: [T2, "A"]
S3: [T2, "A"]
S4: (空)
S5: (空)
```

> ⚠️ **危险信号：** 日志`[T2, "A"]`已存在于3台服务器上，达到了"过半数"

**第2步：任期3**
1. S5发起新选举，进入任期3
2. S5获得S4和S3的投票，成功当选Leader
3. S5收到客户端请求，写入数据"B"
4. S5在日志中追加`[T3, "B"]`
5. S5也崩溃了！没来得及同步给其他服务器

```
当前状态:
S1: [T2, "A"] (已崩溃)
S2: [T2, "A"]
S3: [T2, "A"]
S4: (空)
S5: [T3, "B"] (已崩溃)
```

**第3步：致命的错误提交**
1. S2发起选举，进入任期4
2. S2获得S3和S4的投票，当选Leader
3. **错误决定：** S2发现`[T2, "A"]`已存在于过半数服务器，于是提交了这条日志
4. 在S2和S3的视角里，数据"A"已经被永久确认！

**第4步：数据被推翻**
1. S5恢复，发起选举进入任期5
2. 投票检查：S5的日志任期3 > S2、S3的最新日志任期2
3. S2和S3必须投票给S5（因为任期更高）
4. S5当选Leader，强制所有Follower与自己的日志保持一致
5. **灾难结果：** S2和S3之前"永久提交"的`[T2, "A"]`被覆盖成`[T3, "B"]`！

##### Raft的解决方案

**核心原则：** Leader永远不能仅凭"数量过半"就提交之前任期的日志，只能通过提交自己当前任期的日志来间接导致之前任期的日志被提交。

**解决机制：**
1. **间接提交原则：** 必须接收到一个新的客户端请求，成功追加到日志并复制给过半数服务器，证实了自己Leader身份
2. **任期号保留：** 当Leader复制之前任期的日志时，Raft会为所有日志保留原始的任期号
3. **日志一致性修复：** 通过递减nextIndex找到共同祖先，确保日志复制的一致性

**日志修复示例：**
```
Leader节点有：1:a, 2:b, 3:d
Follower节点有：1:a, 2:c

修复过程：
1. 通过递减nextIndex找到共同祖先1:a
2. Follower丢弃1:a之后的内容
3. 开始复制Leader的内容
4. 最终达到一致性
```

##### 故障处理机制

**跟随者和候选者崩溃：**
- Raft通过无限重试处理崩溃
- 由于RPCs都是幂等的，重试不会造成问题
- 如果跟随者收到已包含的日志，会直接忽略请求

##### 时间和可用性要求

Raft要求安全性不能依赖时间，但为了保证可用性，需要满足以下时间条件：

```
广播时间 << 选举超时时间 << 平均故障间隔时间
```

**参数说明：**
- **广播时间：** 服务器发送给其他服务器并接收响应的平均时间
- **选举超时时间：** 触发新选举的等待时间
- **平均故障间隔时间（MTBF）：** 一台服务器两次故障之间的平均时间

**实际数值参考：**
- 广播时间：0.5-20毫秒（取决于存储技术）
- 选举超时时间：10-500毫秒
- 平均故障间隔时间：几个月甚至更长

> 💡 **关键洞察：** 广播时间必须比选举超时时间小一个量级，这样才能稳定发送心跳来防止跟随者进入选举状态。
